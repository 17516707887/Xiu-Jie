# 生活助手 AI Agent 完整测试用例文档

## 一、测试概述

### 1.1 测试目的
验证生活助手 AI Agent 的功能完整性、上下文记忆准确性、脚本可用性、异常容错能力，确保核心场景可正常运行，边界场景有友好的异常处理。

### 1.2 测试范围
覆盖项目所有核心功能：
• 基础工具功能（天气查询、快递费计算、日期查询、购物清单、出行建议）；
• 上下文记忆能力（多轮对话延续）；
• 一键启动脚本（start.sh）的完整性与鲁棒性；
• 异常场景（缺失 API Key、无效参数、端口占用等）的容错能力。

## 二、功能测试用例

### 2.1 天气查询功能
用例 1：基础城市天气查询
• 测试场景：查询指定城市的实时天气
• 测试步骤：
1. 启动 Agent（命令行 / Web 端）；
2. 输入指令：北京天气；
• 预期结果：
◦ 返回北京实时天气信息（温度、湿度、风力、天气状况）；
◦ 无报错信息，格式清晰。

用例 2：模糊天气查询（带时间）
• 测试场景：查询指定城市指定时间的天气
• 测试步骤：
1. 启动 Agent；
2. 输入指令：上海明天天气；
• 预期结果：
◦ 返回上海次日的天气预报；
◦ 上下文记忆中记录 “last_city = 上海”“last_topic=weather”。

### 2.2 快递费计算功能
用例 ：基础快递费计算
• 测试场景：输入重量和距离，计算快递费
• 测试步骤：
1. 启动 Agent；
2. 输入指令：计算5kg行李、800km距离的快递费；
• 预期结果：
◦ 按 “基础费 8 元 + 重量费（5×2=10 元）+ 距离费（800/100*0.5=4 元）” 核算，返回总费用 22 元及明细；
◦ 上下文记忆中记录 “last_weight=5”“last_distance=800”。

### 2.3 日期查询功能
用例 ：基础日期查询
• 测试场景：查询今天 / 明天 / 昨天的日期与星期
• 测试步骤：
1. 启动 Agent；
2. 依次输入：今天星期几、明天日期、昨天是几号；
• 预期结果：
◦ 准确返回对应日期与星期，格式如 “2026 年 3 月 5 日 星期三”；
◦ 无时间计算错误。

### 2.4 购物清单管理功能
用例 1：清单基础操作（添加 + 查看）
• 测试场景：添加商品并查看清单
• 测试步骤：
1. 启动 Agent；
2. 输入：添加苹果到购物清单；
3. 输入：查看购物清单；
• 预期结果：
◦ 步骤 2 返回 “✅ 已添加「苹果」到购物清单”；
◦ 步骤 3 返回 “🛒 购物清单：1. 苹果”。

用例 2：清单高级操作（移除 + 清空）
• 测试场景：移除商品并清空清单
• 测试步骤：
1. 启动 Agent；
2. 输入：添加牛奶、添加面包；
3. 输入：移除牛奶；
4. 输入：清空购物清单；
5. 输入：查看清单；
• 预期结果：
◦ 步骤 3 返回 “✅ 已从购物清单移除「牛奶」”；
◦ 步骤 4 返回 “✅ 购物清单已清空”；
◦ 步骤 5 返回 “🛒 购物清单：暂无商品”。

### 2.5 出行建议生成功能
用例 ：基础出行建议
• 测试场景：结合城市、重量、距离生成出行建议
• 测试步骤：
1. 启动 Agent；
2. 输入：去广州带10kg行李，路程1500km，后天出发，给出行建议；
• 预期结果：
◦ 先查询广州后天天气；
◦ 结合天气、行李重量、距离生成建议（如 “携带防晒霜、10kg 行李建议托运”）；
◦ 结果整合为自然语言，无分条罗列。

## 三、上下文记忆测试用例

### 3.1 天气话题延续
• 测试场景：查询天气后追问 “明天呢”，验证上下文延续
• 测试步骤：
1. 启动 Agent；
2. 输入：深圳天气；
3. 输入：明天呢；
4. 输入：后天呢；
• 预期结果：
◦ 步骤 2 返回深圳当天天气，记录 “last_city = 深圳”“last_tool_called=get_weather”；
◦ 步骤 3 返回深圳明天天气（而非日期）；
◦ 步骤 4 返回深圳后天天气，无需要重新输入城市。

## 四、异常场景测试用例

### 4.1 缺失 API Key 测试
• 测试场景：未配置 DASHSCOPE_API_KEY/AMAP_KEY
• 测试步骤：
1. 清空.env 文件中的 API Key；
2. 启动 Agent；
3. 输入：杭州天气；
• 预期结果：
◦ 启动时提示 “⚠️ 警告：未配置 API Key，将使用模拟模式”；
◦ 步骤 3 返回模拟天气数据（如 “🌤️ 杭州：晴，25℃，湿度 60%”）；
◦ 无程序崩溃，功能降级可用。

### 4.2 无效参数测试
• 测试场景：输入非数字的重量 / 距离
• 测试步骤：
1. 启动 Agent；
2. 输入：计算abc公斤 xyz公里的快递费；
• 预期结果：
◦ 识别参数无效，提示 “😓 未识别到有效的重量 / 距离，请输入数字（如 2kg 300km）”；
◦ 无语法错误 / 程序崩溃。

## 五、测试结果判定标准

### 5.1 功能测试通过标准
• 所有核心工具功能可正常执行，返回结果准确；
• 上下文记忆符合预期，延续性问题无需重复输入关键参数；
• 结果整合为自然语言，无工具调用痕迹。

### 5.2 异常测试通过标准
• 所有异常场景有友好提示，无程序崩溃；
• 功能可降级使用（如无 API Key 时返回模拟数据）；
• 错误提示明确，可指导用户解决问题。

## 六、测试总结

### 6.1 核心测试点
1. 基础工具功能的准确性（天气 / 快递费 / 日期 / 清单 / 出行建议）；
2. 上下文记忆的智能性（延续性问题的正确响应）；
3. 一键脚本的便捷性（路径 / 权限 / 进程管理）；
4. 异常场景的容错性（参数 / API / 端口 / 编码错误）。

### 6.2 测试优先级
• 高优先级：基础工具功能、上下文记忆、start.sh 脚本启动；
• 中优先级：混合话题上下文、无效参数处理；
• 低优先级：偏移天数查询、清单高级操作。
### 6.3 备注
• 测试前需确保.env 文件配置正确的 API Key；
• 上下文记忆测试需连续执行，不可中途清空记忆；
• 脚本测试需在 Linux/WSL 环境执行，Windows 需借助 Git Bash。
